# Product Principles

- Build tools that remove real operational burden, not demos that only look smart.
- Prefer narrow, high-frequency use cases over broad assistant claims.
- Make the user, buyer, and decision maker explicit.
- Treat trust, consent, and explainability as product features.
- Kill weak ideas quickly and record why.
- Build the first useful workflow before building a full platform.
- Keep the default path AI-first: the platform drafts research, risks, validation, product scope, and handoff materials before asking the user to decide.
- Ask the user for judgment, not busywork. Manual editing is a correction path, not the main workflow.
- Classify the intended product surface early: web app, mobile app, web site, automation, or operator console.
- Treat external development tools as delivery choices layered on top of the product surface. Cursor, Codex, Claude Code, Google Antigravity, and future MCP connectors should change setup files and sync instructions, not the underlying result type.
- Make every downstream artifact serve the final production package: PRD, design direction, stack choice, build scope, and prompts should be ready for native build or selected external development-tool execution.
- Keep the visible console simple enough for a first-time operator: one main action, AI-prepared drafts, optional manual corrections, and bottom-only step advancement.
- Monetize execution depth, not confusion: basic idea understanding and safety guidance stay useful, while deeper market scans, production package export, external tool sync, and team workflows become paid upgrade moments.
- Do not expose internal implementation vocabulary in primary UI copy. Use product-facing terms such as result type, production package, execution document, market check, risk check, and first build scope.
- When several artifacts are needed, let AI prepare them as one guided sequence and ask the user for one clear confirmation instead of several manual save chores.
