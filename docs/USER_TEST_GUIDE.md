# AI Venture Lab 사용 및 테스트 가이드

이 문서는 처음 테스트하는 운영자와 베타 사용자가 AI Venture Lab의 핵심 흐름을 혼동 없이 확인하도록 만든 가이드입니다.

현재 제품 원칙은 `docs/AI_FIRST_PRODUCT_FLOW.md`와 `docs/PRODUCT_PRINCIPLES.md`를 따릅니다. 사용자는 정리되지 않은 아이디어, 회의 메모, GPT 대화, 자동화하고 싶은 업무를 넣고, AI가 먼저 사업성, 리스크, 시장성, 검증 계획, 제작 패키지를 정리합니다. 사용자는 중요한 판단만 확인하고 저장합니다.

## 1. 기본 원칙

처음 테스트할 때는 아래 원칙만 기억합니다.

1. 사용자는 처음부터 세부 양식을 채우지 않습니다.
2. `아이디어 도출`에서 메모를 붙여넣고 `이 내용으로 아이디어 정리하기`만 누릅니다. 아이디어가 아예 없을 때만 `AI가 아이디어 도출하기`를 사용합니다.
3. AI가 만든 요약과 결과물 형태를 확인한 뒤 저장합니다.
4. 각 단계는 저장이 끝난 뒤 하단 `다음` 버튼으로만 이동합니다.
5. 복사, 채우기, 보조 버튼은 선택 기능이며 단계를 이동시키지 않습니다.
6. 제작 단계에서는 `AI 제작 패키지 만들기`와 `제작 패키지 저장`만 기본 흐름으로 봅니다.

## 2. 접속 전 확인

운영 URL: `https://ai-venture-lab.vercel.app`

기본 테스트에 필요한 준비는 세 가지입니다.

| 확인 항목 | 기준 |
| --- | --- |
| 테스트 계정 | Supabase Auth에 등록되어 있고 이메일 확인이 완료된 계정 |
| 워크스페이스 | 로그인 후 활성 워크스페이스가 보이거나 `워크스페이스 만들기`로 생성 가능 |
| AI 환경변수 | Production에 `OPENAI_API_KEY`가 등록되어 있으면 AI 정리가 우선 실행됨 |

비밀번호 로그인 테스트를 기본으로 사용합니다. 이메일 링크 로그인은 SMTP 설정과 발송 제한의 영향을 받을 수 있으므로 보조 옵션입니다.

워크스페이스 생성 중 `organizations` RLS 오류가 보이면 Supabase SQL Editor에서 `supabase/migrations/20260512010000_repair_workspace_creation_policy.sql` 내용을 실행한 뒤 새로고침합니다.

## 3. 빠른 성공 경로

아래 순서가 베타 사용자에게 보여줄 기본 흐름입니다.

처음에는 이 다섯 줄만 기억하면 됩니다.

1. 메모 붙여넣기
2. 아이디어 저장
3. 사업성 평가 저장
4. 검증 자료 저장
5. 제작 패키지 저장 후 최종 실행으로 넘기기

1. 운영 URL을 엽니다.
2. 로그인합니다.
3. 실행 보드에서 `아이디어 도출`을 엽니다.
4. 입력칸에 회의 메모, 아이디어, GPT 대화, 자동화하고 싶은 업무를 그대로 붙여넣습니다.
5. 입력칸에 내용을 붙여넣었으면 `이 내용으로 아이디어 정리하기`를 누릅니다. 입력할 아이디어가 없으면 `AI가 아이디어 도출하기` 하나만 눌러 후보를 먼저 만듭니다.
6. AI가 정리한 아이디어, 구매자, 대상 사용자, 결과물 형태, 검증 질문을 확인합니다.
7. `이 아이디어 저장하고 검증 시작` 또는 화면에 표시되는 저장 버튼을 누릅니다.
8. 하단 `다음: 사업성 평가` 버튼이 활성화되면 다음 단계로 이동합니다.
9. 사업성 평가를 확인하고 `점수 저장`을 누릅니다.
10. 하단 `다음: 검증 계획 저장` 버튼으로 STEP 3에 이동합니다.
11. `AI 추천 검증 계획 저장`을 누릅니다.
12. `시장·경쟁 자동 점검` 결과가 저장 완료 상태인지 확인합니다.
13. 하단 `다음: 검증 자료 저장` 버튼으로 이동합니다.
14. 검증 자료 저장 상태를 확인합니다.
15. `AI 제작 패키지 만들기`를 누릅니다.
16. AI가 진행 상태를 보여준 뒤 제작 패키지 요약을 표시하는지 확인합니다.
17. 필요한 경우 보완 메모만 추가합니다.
18. `제작 패키지 저장`을 누릅니다.
19. 제작 패키지 저장 후에는 최종 실행 단계가 열리고, 패키지 파일 받기는 최종 실행에서만 보이는지 확인합니다.

이 흐름을 통과하면 첫 사용자 기준의 핵심 테스트는 완료입니다.

## 4. 샘플 입력

아래 문장을 그대로 붙여넣어 빠르게 테스트할 수 있습니다.

```text
아이디어: 요양보호사 운영 콘솔

방문요양센터장은 요양보호사 일정, 보호자 문의, 돌봄 기록, 변경 요청을 카카오톡과 엑셀로 따로 관리하고 있다.
일정 누락이나 기록 누락이 생기면 보호자 문의가 반복되고, 센터장은 매일 확인에 시간을 쓴다.

구매자는 소규모 방문요양센터장이고, 실제 사용자는 센터 운영자, 요양보호사, 보호자다.
처음에는 일정 공유, 방문 기록, 보호자 알림, 누락 체크만 작게 만들고 싶다.
개인 건강정보나 의료 판단은 MVP 범위에서 제외해야 한다.
```

다른 도메인을 테스트하고 싶다면 아래 샘플도 사용할 수 있습니다.

```text
아이디어: 구독 정리 도우미

카드 명세서와 이메일에 흩어진 반복 결제를 찾아서 사용자가 잊고 있던 구독을 정리해주는 서비스.
구매자는 디지털 구독이 많은 개인 사용자이고, 반복 결제 누락과 해지 번거로움이 문제다.
처음에는 카드 명세서 CSV와 이메일 제목을 사용자가 직접 업로드하고, 해지 링크와 절약 예상액을 보여주는 웹 서비스로 검증한다.
```

## 5. 단계별 기대 상태

### STEP 1. 아이디어 도출

기대 상태:

| 확인 항목 | 기대 결과 |
| --- | --- |
| 입력 전 버튼 | `AI가 아이디어 도출하기` 하나가 활성 상태 |
| 입력 후 버튼 | `이 내용으로 아이디어 정리하기` 하나가 활성 상태 |
| 후보 도출 후 버튼 | `킵한 후보로 아이디어 정리하기`와 `다른 후보 더 확인하기`가 같은 위치에 표시 |
| 실행 버튼 수 | 상태별로 필요한 실행 버튼만 표시 |
| 결과물 형태 | 웹 서비스, 모바일 앱, 랜딩/웹사이트, 업무 자동화, 운영 콘솔 중 하나로 정리 |
| 저장 후 이동 | 저장 전에는 다음 단계 버튼이 비활성, 저장 후 활성 |

`필요할 때만 AI 정리 다시 보기`는 결과가 어색하거나 빠진 후보가 있어 보일 때만 엽니다. 후보 비교, 규칙 기반 감사, 상위 후보 일괄 저장은 운영자 검수용입니다. 첫 사용자 테스트에서는 필수로 누르지 않습니다.

### STEP 2. 사업성 평가

기대 상태:

| 확인 항목 | 기대 결과 |
| --- | --- |
| 평가 항목 | 문제 강도, 빈도, 도달 가능성, 지불 의향, MVP 속도, 차별성, 리스크 감점 표시 |
| 저장 버튼 | 점수 저장 후 완료 상태로 바뀌고 중복 클릭을 유도하지 않음 |
| 다음 단계 | 저장 완료 후 하단 `다음` 버튼이 활성 |
| 결과물 형태 반영 | 사업성 해석과 제작 난이도 설명에 STEP 1의 결과물 형태가 반영 |

### STEP 3. 검증 계획과 시장 점검

기대 상태:

| 확인 항목 | 기대 결과 |
| --- | --- |
| AI 추천 검증 계획 | 한 번 저장하면 `검증 계획 저장 완료` 상태 |
| 시장·경쟁 자동 점검 | 수요, 경쟁도, 포화도, 진입장벽, 대체재, 참고 출처가 AI 준비 자료로 표시 |
| 선택 입력 | 실제 인터뷰나 테스트 결과가 있을 때만 펼쳐서 저장 |
| 다음 단계 | 검증 계획과 시장 점검 저장이 끝난 뒤 활성 |

질문 복사, 입력칸 채우기, 보강 질문은 선택 기능입니다. 처음 테스트에서는 AI 추천 저장과 시장 점검 저장 상태만 확인합니다.

### STEP 4. 검증 자료 저장

기대 상태:

| 확인 항목 | 기대 결과 |
| --- | --- |
| 검증 자료 | 아이디어 요약, 조사 요약, 7일 검증 계획, 검증 완료 요약이 정리됨 |
| 저장 상태 | 저장 완료 후 버튼이 완료 상태로 바뀜 |
| 다음 단계 | 저장 완료 후에만 제작 패키지 단계가 열림 |

### STEP 5. 제작 패키지

기대 상태:

| 확인 항목 | 기대 결과 |
| --- | --- |
| 기본 실행 | `AI 제작 패키지 만들기` 하나로 제작 자료 정리 시작 |
| 진행 표시 | 검증 결과 읽기, 결과물 형태 확인, 제작 범위 정리, 디자인/기술 방향 정리, 외부 개발 도구 전달 자료 정리 상태가 순서대로 보임 |
| 요약 화면 | PRD, IA, 디자인 방향, 기술 스택, 작업 순서, 검증 근거, 제작 지시가 한 화면에 요약 |
| 보완 메모 | 필요한 경우만 추가 |
| 저장 | `제작 패키지 저장` 후 완료 상태로 바뀜 |
| 후속 버튼 | 저장 완료 후에만 다음 단계가 열림. 실제 파일 받기는 최종 실행 단계에서만 보임 |

제작 패키지에는 STEP 1/2에서 정한 결과물 형태가 반영되어야 합니다. 예를 들어 운영 콘솔이면 IA와 기술 스택이 권한, 목록, 상태 변경, 감사 기록 중심으로 정리되어야 하고, 랜딩/웹사이트면 전환 문구, 섹션 구조, 폼 수집, 성과 측정 중심으로 정리되어야 합니다.

## 6. 제작 패키지에서 확인할 내용

저장된 제작 패키지는 단순 문서가 아니라 외부 개발 도구가 바로 읽을 수 있는 제작 자료여야 합니다.

| 패키지 항목 | 확인 기준 |
| --- | --- |
| PRD | 대상 사용자, 구매자, 해결 문제, MVP 범위, 비범위가 분명함 |
| IA | 결과물 형태에 맞는 화면 구조와 주요 상태가 있음 |
| 디자인 방향 | 사용자 도메인에 맞는 UI 밀도, 문구 톤, 주요 컴포넌트가 있음 |
| 기술 스택 | 결과물 형태, 인증, 데이터, 보안, 배포 조건에 맞게 정리됨 |
| 작업 순서 | 바로 시작할 첫 작업, 선행 조건, 완료 기준이 있음 |
| 검증 근거 | 사업성 평가, 시장 점검, 리스크, 검증 계획이 연결됨 |
| 외부 도구 전달 자료 | 외부 제작 도구가 읽을 제작 지시와 검증 기준이 있음 |

사용자 화면에서는 내부 실행 구조보다 `제작 패키지`, `제작 자료`, `작업 순서`, `검증 자료`, `기획서`처럼 보이는지 확인합니다.

## 7. 최종 실행과 외부 제작 도구 연결

최종 실행 단계는 문서를 내려받는 곳이 아니라, 검증된 제작 패키지를 실제 제작 환경으로 넘기는 마지막 연결 단계입니다.

Cursor, Codex, Claude Code, Google Antigravity 중 하나를 선택한 경우 기대 상태:

| 확인 항목 | 기대 결과 |
| --- | --- |
| 제작 방식 | `외부 제작 도구로 개발`과 선택한 도구 이름이 표시됨 |
| 연결 파일 | 선택한 도구 이름의 `연결 파일 받기`가 보이고, 프로젝트 루트에서 실행할 PowerShell 명령이 표시됨 |
| 설치 파일 | Cursor는 `.cursor`, Codex는 `.codex`, Claude Code는 `.claude`와 `.mcp.json`, Google Antigravity는 `.antigravity` 파일을 만든다고 안내됨 |
| 시작 지시 | 선택한 도구에 맞는 `AI_VENTURE_*_START.md` 파일을 첫 작업 메시지로 넣으라고 안내됨 |
| 작업 반영 | Cursor는 `venture_record_progress`, 나머지 도구는 각 도구 폴더의 `venture-lab-cli.mjs record-progress`로 로컬 기록과 Venture Lab 작업 상태를 함께 업데이트한다고 안내됨 |
| 자동 확인 | 최종 실행 화면이 저장된 작업 상태를 주기적으로 다시 읽고, 작업 목록과 성과 확인 화면에 반영한다고 안내됨 |
| 백업 경로 | 자동 반영 실패 시에만 `.cursor`, `.codex`, `.claude`, `.antigravity`의 `venture-lab-progress.json`을 백업 가져오기에 붙여넣는다고 안내됨 |

운영 환경에는 `SUPABASE_SERVICE_ROLE_KEY`가 있어야 합니다. `BUILD_SYNC_TOKEN_SECRET`은 선택값이며, 설정하면 외부 제작 도구 연결 토큰 서명에 별도 비밀값을 사용할 수 있습니다. `public.build_sync_tokens` 마이그레이션과 `20260525020000_allow_named_build_sync_tools.sql`이 적용되어야 최종 실행 화면에서 네 도구의 연결 목록, 최근 사용 시각, 개별 연결 끊기를 확인할 수 있습니다.

사용자 테스트 흐름:

1. 최종 실행 단계에서 개발 도구가 Cursor, Codex, Claude Code, Google Antigravity 중 하나인지 확인합니다.
2. 선택한 도구의 `연결 파일 받기`를 눌러 PowerShell 파일을 받습니다.
3. 받은 파일을 실제 개발할 프로젝트 루트로 옮깁니다.
4. 선택한 개발 도구에서 그 프로젝트 폴더를 엽니다.
5. 터미널 또는 PowerShell에서 화면에 표시된 명령을 실행합니다.
6. 화면에 나온 `node .*/venture-lab-cli.mjs next-task` 명령을 실행해 첫 작업이 읽히는지 확인합니다.
7. Cursor와 Claude Code는 MCP 연결에서 `ai-venture-lab` 서버가 보이는지 확인합니다. Codex와 Antigravity는 로컬 CLI가 정상 실행되는지 확인합니다.
8. 선택한 도구에 맞는 `AI_VENTURE_*_START.md` 내용을 첫 메시지로 붙여 넣고 첫 작업을 시작합니다.
9. 작업이 끝나면 `venture_record_progress` 또는 `node .*/venture-lab-cli.mjs record-progress ...`로 완료 보고를 남기게 합니다.
10. Venture Lab 최종 실행 화면을 새로고침해 작업 상태가 반영됐는지 확인합니다.

이 단계에서 사용자가 직접 JSON을 붙여넣는 흐름은 기본 흐름이 아닙니다. 백업 가져오기는 접힌 보조 경로여야 하며, 자동 반영이 실패했을 때만 사용합니다.

## 8. STEP 8 성과 확인

성과 확인 화면은 기본적으로 `지금 할 일`, 제작 작업 진행표, 다음 판단만 먼저 봅니다. T-001, T-002처럼 작업별 완료/진행/차단 상태가 리스트로 보여야 하며, 사용자는 다음 작업을 이어서 할지, 출시 신호를 연결할지만 판단하면 됩니다.

| 확인 항목 | 기대 결과 |
| --- | --- |
| 지금 할 일 | 상태에 따라 `최종 실행으로 가기`, `리포트 복사` 같은 단일 CTA가 보임 |
| 작업 진행표 | 작업별 완료/진행/차단 상태와 다음 작업이 리스트로 보임 |
| 빈 상태 | 성과 확인은 아직 볼 게 없다고 말하고 `최종 실행 열기` 버튼을 제공 |
| 운영자 리포트 | 리포트 복사/저장, 상세 이벤트 수집 주소, 퍼널 리포트는 `필요할 때만 여는 운영자용 리포트` 안에 접혀 있음 |

## 9. 목록 관리

아이디어를 정리할 때는 목록 화면에서 한 번에 처리할 수 있습니다.

| 화면 | 기대 결과 |
| --- | --- |
| 검토 아이디어 | 체크박스로 여러 아이디어를 선택하고 `선택한 아이디어 삭제` 또는 `검토 아이디어 전체 삭제`로 삭제 목록에 옮김 |
| 삭제한 아이디어 | 체크박스로 여러 아이디어를 선택하고 `선택한 아이디어 되살리기` 또는 `선택한 아이디어 완전 삭제`를 실행 |
| 완전 삭제 | 연결된 리스크, 실험, 제작 자료, 작업, 이벤트 기록을 먼저 정리한 뒤 아이디어를 삭제하며 확인창이 반드시 표시 |

완전 삭제는 되돌릴 수 없습니다. 베타 테스트 중에는 먼저 삭제 목록으로 옮긴 뒤, 정말 필요할 때만 완전 삭제를 실행합니다.

## 10. 고급 운영자 검수

아래 항목은 첫 사용자에게 요구하지 않습니다. 배포 전 운영자가 넓은 회귀 범위를 확인할 때만 사용합니다.

| 고급 검수 항목 | 언제 확인하는지 |
| --- | --- |
| AI 정리 다시 보기, 후보 비교, 규칙 기반 감사 | AI 후보 품질과 폴백을 비교할 때 |
| 상위 후보 일괄 저장 | 같은 원문에서 여러 후보 저장을 검증할 때 |
| 산출물 라이브러리 승인 큐 | PRD, MVP 명세, 디자인 브리프, 기술 명세 승인 상태를 점검할 때 |
| 실행 할 일, 직접 태스크 추가 | 제작 패키지 이후 실제 구현 태스크를 운영할 때 |
| 완료 보고, QA 매트릭스, 빌드 명령 패킷 | MVP 제작 완료 후 출시 판단을 점검할 때 |
| 텔레메트리 퍼널 | 외부 MVP 이벤트 수집과 학습 리포트를 점검할 때 |

보조 기능을 눌렀을 때도 단계가 자동 이동하면 안 됩니다. 단계 이동은 항상 저장 완료 후 하단 다음 버튼으로만 일어납니다.

## 11. 텔레메트리 운영 스모크

텔레메트리 수집 API는 외부 MVP나 운영 서버가 이벤트를 보낼 때 사용합니다. 일반 사용자 테스트에서는 실행하지 않습니다.

필요한 Production 환경변수:

| 환경변수 | 위치 | 용도 |
| --- | --- | --- |
| `SUPABASE_SERVICE_ROLE_KEY` | Vercel 서버 전용 | 수집 API가 검증된 이벤트를 저장 |
| `TELEMETRY_INGEST_SECRET` | Vercel 서버 전용, 외부 MVP 서버에도 서버 전용으로 보관 | 이벤트 수집 API 호출 인증 |

운영 스모크:

```powershell
$env:TELEMETRY_INGEST_SECRET="Vercel에 등록한 TELEMETRY_INGEST_SECRET"
$env:TELEMETRY_SMOKE_IDEA_ID="public.ideas.id에서 복사한 테스트 idea id"
pnpm smoke:telemetry
pnpm smoke:telemetry:funnel
```

`TELEMETRY_SMOKE_IDEA_ID`는 `auth.users.id`가 아니라 `public.ideas.id`입니다. 비밀값은 코드나 문서에 저장하지 말고 현재 터미널 환경변수 또는 로컬 env 파일에만 둡니다.

이벤트에는 이메일, 전화번호, 이름, 계좌, 카드, 원문 대화 같은 직접 식별 정보를 넣지 않습니다.

## 12. 배포 전 검증 명령

제품 동작, 제작 패키지, 배포 흐름을 바꿨다면 아래를 기준으로 검증합니다.

```powershell
pnpm quality:full
pnpm smoke:prod
pnpm smoke:routes
pnpm smoke:market
pnpm smoke:browser
pnpm smoke:browser:auth
pnpm smoke:build-sync
```

`pnpm smoke:market`은 프로덕션 시장·경쟁 자동 점검 API가 웹 검색 포함 모드로 응답하는지 확인합니다. 기본 제한 시간은 웹 검색 지연을 감안해 180초입니다. OpenAI 또는 웹 검색이 일시적으로 불안정한 상황에서 로컬 추정 응답까지 허용하려면 현재 터미널에 `MARKET_SCAN_SMOKE_ALLOW_ESTIMATE=1`을 설정한 뒤 실행합니다.

인증된 쓰기 스모크가 필요할 때만 아래처럼 실행합니다. 이 명령은 실제 테스트 아이디어를 만들 수 있으므로 베타 전용 계정에서만 사용합니다.

```powershell
$env:BROWSER_SMOKE_ALLOW_WRITE="1"
pnpm smoke:browser:auth
```

RLS 경계를 다시 확인할 때는 사전에 보존한 테스트 계정과 아이디어가 있는지 확인한 뒤 실행합니다.

```powershell
pnpm smoke:browser:rls:preflight
pnpm smoke:browser:rls
```

## 13. 완료 기준

첫 사용자 기준 완료:

1. 로그인 후 실행 보드에 들어갈 수 있다.
2. 아이디어 메모를 붙여넣으면 AI가 하나의 검토 후보와 결과물 형태를 정리한다.
3. 저장 전에는 다음 단계가 열리지 않는다.
4. 사업성 평가 저장 후 버튼이 완료 상태로 보인다.
5. 검증 계획과 시장·경쟁 점검이 AI 준비 자료로 저장된다.
6. 제작 패키지가 한 번의 AI 실행과 한 번의 저장으로 만들어진다.
7. 제작 패키지에 PRD, IA, 디자인 방향, 기술 스택, 작업 순서, 검증 근거, 외부 개발 도구 전달 자료가 포함된다.
8. 제작 패키지 저장 후 최종 실행 단계가 열린다. 파일 다운로드는 최종 실행에서만 보인다.
9. Cursor, Codex, Claude Code, Google Antigravity 중 선택한 도구의 연결 파일을 받으면 외부 프로젝트에서 작업 상태 자동 반영을 시작할 수 있다.

운영자 기준 완료:

1. `pnpm quality:full`이 통과한다.
2. 프로덕션 기본, 라우트, 브라우저, 인증 스모크가 통과한다.
3. 필요 시 RLS 허용/차단 스모크와 텔레메트리 스모크가 통과한다.
4. 변경된 결정은 `docs/DECISION_LOG.md`에 기록된다.
5. 완료한 큐는 커밋, 푸시, 배포 상태, 검증 결과와 함께 보고된다.
