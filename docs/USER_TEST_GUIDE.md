# AI Venture Lab 사용 및 테스트 가이드

이 문서는 AI Venture Lab을 처음 테스트하는 운영자가 아이디어를 입력하고, 점수화하고, 리스크와 실험을 붙인 뒤, 전략/리서치/제품/디자인/개발/QA/디버깅/보안/출시 오케스트레이션까지 한 번에 검증할 수 있도록 만든 실행 설명서입니다.

## 1. 플랫폼 목적

AI Venture Lab은 앱 아이디어를 바로 개발로 밀어 넣는 도구가 아니라, 아이디어를 MVP 후보로 승격하기 전에 필요한 판단 근거를 쌓는 실행 센터입니다.

핵심 흐름은 다음과 같습니다.

1. 대화나 메모에서 아이디어 후보를 자동 발굴하거나 아이디어를 짧게 접수합니다.
2. 문제 강도, 빈도, 지불 의향, MVP 속도, 차별성, 리스크 감점을 점수화합니다.
3. 리스크, 판단 기록, 실험 계획과 실험 결과 학습을 붙입니다.
4. 전략부터 출시까지 오케스트레이션 런북을 생성합니다.
5. 아이디어 브리프, 리서치 브리프, 검증 완료 요약, PRD, MVP 명세, 백엔드 결정, 디자인 브리프, 기술 명세, 개발 런북, 출시 체크리스트 산출물을 저장하고 승인합니다.
6. 출시 준비도 게이트가 통과되는지 확인합니다.

## 2. 첫 접속 체크리스트

운영 URL: `https://ai-venture-lab.vercel.app`

처음 테스트할 때는 아래 순서대로 확인합니다.

1. 페이지가 열리는지 확인합니다.
2. `운영자 로그인` 영역에 이메일을 입력합니다.
3. `매직 링크 보내기`를 클릭합니다.
4. 이메일 받은 편지함에서 Supabase 로그인 링크를 클릭합니다.
5. 다시 앱으로 돌아왔을 때 `로그인됨`과 이메일 주소가 보이면 성공입니다.
6. `워크스페이스 상태`에서 워크스페이스가 없으면 `워크스페이스 만들기`를 클릭합니다.
7. 기존에 만든 개인 기록이 있다면 `개인 기록 연결`로 현재 워크스페이스에 붙입니다.

참고: 매직 링크 로그인은 비밀번호가 필요 없습니다. 비밀번호 로그인은 Supabase Auth에서 비밀번호 기반 사용자로 따로 만든 계정에만 사용합니다.

## 3. 테스트용 샘플 아이디어

아래 샘플을 그대로 입력하면 전체 흐름을 빠르게 검증할 수 있습니다.

### 샘플 A: 요양보호사 운영 콘솔

| 입력 필드 | 예시 값 |
| --- | --- |
| 이름 | 요양보호사 운영 콘솔 |
| 구매자 | 방문요양센터장, 가족 돌봄 관리자 |
| 한 줄 설명 | 가족, 센터, 요양보호사의 일정과 돌봄 기록을 한 화면에서 관리하는 운영 콘솔 |
| 대상 사용자 | 소규모 방문요양센터 운영자와 보호자 |
| 수요 신호 | 장기요양 수요 증가, 보호자와 센터 간 커뮤니케이션 누락, 돌봄 기록 확인 요구가 반복적으로 발생 |
| 리스크 요약 | 건강정보와 개인정보 처리, 요양보호사 근무 기록의 법적 책임, 가족 간 권한 관리 이슈 |
| 다음 증거 | 센터장 5명 인터뷰, 보호자 10명 설문, 기존 업무표/카카오톡 운영 방식 수집 |

### 샘플 B: 관계 선물 에이전트

| 입력 필드 | 예시 값 |
| --- | --- |
| 이름 | 관계 선물 에이전트 |
| 구매자 | 바쁜 직장인, 영업/컨설팅 직군 |
| 한 줄 설명 | 중요한 사람의 기념일과 취향을 기억하고 선물 후보와 메시지를 추천하는 관계 관리 도구 |
| 대상 사용자 | 가족, 파트너, 고객 관계를 자주 챙겨야 하는 30-40대 전문직 |
| 수요 신호 | 생일/기념일을 놓쳐 관계 부담이 생기고, 센스 있는 선물 선택에 시간이 많이 듦 |
| 리스크 요약 | 개인정보와 연락처 처리, 과도한 자동화가 관계의 진정성을 해칠 수 있음 |
| 다음 증거 | 유료 선물 추천 서비스 사용 경험 인터뷰, 반복 구매 빈도, 선물 예산 범위 조사 |

## 4. 새 아이디어 입력 방법

`아이디어 발굴` 탭을 쓰면 Gemini/ChatGPT 대화, 회의록, 메모에서 후보를 추출할 수 있습니다.

테스트 순서:

1. `아이디어 발굴`을 엽니다.
2. 대화 원문을 붙여넣거나 `샘플 넣기`를 클릭합니다.
3. `AI 후보 발굴`을 클릭합니다. 서버에 `OPENAI_API_KEY`가 있으면 OpenAI Responses API의 구조화 출력으로 후보를 만들고, 키가 없거나 호출이 실패하면 자동으로 규칙 기반 엔진으로 전환됩니다.
4. 같은 원문을 두 방식으로 감사하고 싶으면 `비교 리플레이`를 클릭합니다. 규칙 기반 후보와 AI 후보를 동시에 재실행해 `공통 후보`, `AI 단독`, `규칙 단독`으로 나눠 보여줍니다.
5. AI 키 없이 로컬 규칙만 테스트하려면 `규칙 기반`을 클릭합니다.
6. `최근 추출 실행`에서 사용 엔진, 모델, 입력 길이, 후보 수, 실행 메모를 확인합니다.
7. `추출 리플레이 비교`가 표시되면 공통 후보를 우선 검토하고, AI 단독 후보는 과잉 추론 여부를, 규칙 단독 후보는 원문 라벨/키워드에만 의존한 후보인지 확인합니다.
8. 후보의 검증 점수, 사업/개발 스코어카드, 리스크 등급, 핵심 가설, 검증 질문, 7일 실험, 중단 기준을 확인합니다.
9. `게이트 판정`을 확인합니다. 판정은 아래 기준으로 저장 전 다음 행동을 제안합니다.
10. `후보 비교 매트릭스`에서 진행 후보, 추가 조사, 전환 검토, 중단 후보의 분포와 실행 순서를 확인합니다.
11. `실행 요약 복사`를 누르면 리플레이 비교와 후보별 게이트, 검증 점수, 준비도, 중복 신호, 다음 행동을 Markdown으로 복사할 수 있습니다.
12. `리포트 저장`을 누르면 후보 비교 결과, 추출 리플레이, 추출 엔진 메타데이터, 익명화된 원문 근거가 `research_note` 산출물로 저장됩니다. 최근 저장 리포트는 같은 화면에서 다시 복사할 수 있습니다.
13. `기존 유사 기록`이 표시되면 기존 아이디어를 확장할지, 새 후보로 분리할지 먼저 판단합니다.
14. `민감정보` 준비도와 `원문 근거`를 확인합니다. 이메일, 전화번호, 카드, 계좌, 신분 정보 패턴은 저장 산출물에서 자동 익명화됩니다.
15. `입력 폼으로 보내기`를 누르면 새 아이디어 입력란에 검증 메모가 채워집니다.
16. `검증 패키지 저장` 또는 `패키지 저장`을 누르면 아이디어, 핵심 리스크, 7일 실험이 한 번에 저장됩니다.
17. 후보가 여러 개라면 후보 비교 매트릭스의 `상위 N개 저장`을 눌러 중복 신호가 없고 준비도 70% 이상인 진행/추가 조사 후보를 최대 3개까지 한 번에 검증 패키지로 저장합니다.

| 게이트 | 기준 | 다음 행동 |
| --- | --- | --- |
| 진행 후보 | 검증 72점 이상, 준비도 80% 이상, 고위험/중복 없음 | 검증 패키지로 저장하고 워크벤치에서 점수와 실험을 확정 |
| 추가 조사 | 검증 58-71점 또는 준비도 미달 | 부족한 문제 신호, 구매자, 검증 지표, 리스크, MVP 범위를 보강 |
| 전환 검토 | 검증 45-57점, 강한 중복, 대상/구매자/범위 재정의 필요 | 기존 아이디어와 병합하거나 세그먼트와 첫 MVP를 다시 좁힘 |
| 중단 후보 | 검증 44점 이하 또는 핵심 문제/MVP 신호 부족 | 새 증거가 생길 때까지 저장하지 않고 후보를 보류 |

리포트 저장과 검증 패키지 저장은 로그인된 운영자만 사용할 수 있습니다. 리포트는 포트폴리오 판단 근거를 남기는 용도이고, 검증 패키지는 선택한 후보를 실제 워크벤치 아이디어로 전환하는 용도입니다. `상위 N개 저장`은 저장 마찰을 줄이는 빠른 처리 버튼이지만 중복 신호가 있거나 준비도가 낮은 후보는 자동 제외합니다.
최근 추출 실행 메타데이터는 같은 원문을 AI와 규칙 기반으로 비교하거나, 나중에 후보 판단 근거를 감사할 때 사용합니다.
사업/개발 스코어카드는 `시장 신호`, `수익화`, `MVP 난이도`, `도달 채널`, `자동화 레버리지`, `보안 부담`을 계산해 후보가 실제 앱으로 옮겨질 때의 부담을 보여줍니다. 점수가 낮은 항목은 저장 전 보강 질문으로 보면 됩니다.
`기존 유사 기록` 표시는 중복 방지를 위한 운영 신호입니다. 완전히 같은 문제면 기존 아이디어를 업데이트하고, 사용자/구매자/해결 방식이 다르면 별도 후보로 저장합니다.
AI 후보 발굴을 프로덕션에서 사용하려면 Vercel Production 환경변수에 `OPENAI_API_KEY`를 넣습니다. 모델을 바꾸고 싶으면 `OPENAI_IDEA_MODEL`을 추가합니다. 기본값은 비용 효율적인 `gpt-5-mini`입니다.
후보가 여러 개일 때는 먼저 매트릭스 상단의 `진행 후보`와 `추가 조사`를 처리하고, `전환 검토`는 기존 기록 병합 또는 세그먼트 축소 판단을 먼저 내립니다.

화면 오른쪽의 `새 아이디어 입력` 영역을 사용합니다.

| 필드 | 입력해야 할 내용 | 좋은 입력 기준 |
| --- | --- | --- |
| 이름 | 서비스 또는 앱 아이디어 이름 | 10-30자 정도로 짧게 씁니다. |
| 구매자 | 실제로 돈을 내는 사람 또는 조직 | 사용자와 구매자가 다를 수 있으므로 따로 적습니다. |
| 한 줄 설명 | 무엇을 누구에게 해결해주는지 한 문장 | 기능 나열보다 문제와 결과가 보이게 씁니다. |
| 대상 사용자 | 매일 사용할 사람 | 연령, 직무, 상황을 구체화합니다. |
| 수요 신호 | 이 문제가 실제로 있다는 근거 | 인터뷰, 반복 업무, 비용, 검색량, 기존 대체재를 적습니다. |
| 리스크 요약 | 법적/보안/운영/시장 위험 | 개인정보, 결제, 의료/법률 조언, 플랫폼 책임 등을 적습니다. |
| 다음 증거 | 다음에 확인할 검증 행동 | 인터뷰 대상, 실험, 랜딩 페이지, 프로토타입 테스트를 적습니다. |

필수 입력은 `이름`과 `한 줄 설명`입니다. 저장 후 상단 `아이디어` 숫자가 증가하고, 아래 `아이디어 워크벤치` 목록에 새 카드가 보이면 성공입니다.

좌측 `우선 검토 후보`는 점수, 실험, 승인된 제품 산출물, 열린 고위험 리스크, 중단 판단을 함께 계산해 상위 후보 3개를 보여줍니다. `리스크 먼저`, `실험 먼저`, `검증 후보`, `개발 후보` 라벨을 보고 다음 처리 순서를 정합니다.

## 5. 점수화 순서

`아이디어 워크벤치`에서 새로 만든 아이디어를 선택한 뒤 오른쪽 점수 영역을 조정합니다.

각 항목은 0-5점입니다.

| 항목 | 의미 | 0점 | 5점 |
| --- | --- | --- | --- |
| 문제 강도 | 사용자가 겪는 고통의 크기 | 있으면 좋은 수준 | 지금 당장 해결하고 싶은 수준 |
| 발생 빈도 | 문제가 얼마나 자주 생기는지 | 가끔 발생 | 매일 또는 매주 반복 |
| 도달 가능성 | 초기 사용자를 만날 수 있는지 | 접근 경로 없음 | 인터뷰/판매 채널이 명확 |
| 지불 의향 | 돈을 낼 가능성 | 무료여야만 사용 | 비용 절감/수익 증가가 명확 |
| MVP 속도 | 작게 만들어 검증 가능한지 | 복잡한 인프라 필요 | 1-2주 안에 검증 가능 |
| 차별성 | 기존 대체재 대비 이유 | 대체재와 거의 같음 | 명확한 차별 지점 있음 |
| 리스크 감점 | 규제/보안/운영 부담 | 부담 없음 | 출시 전 큰 차단 요인 |

권장 해석:

| 총점 | 추천 판단 |
| --- | --- |
| 22점 이상 | 진행 |
| 15-21점 | 추가 조사 |
| 9-14점 | 전환 |
| 8점 이하 | 중단 |

점수를 조정한 뒤 `점수 저장`을 클릭합니다. 추천 판단은 참고용이며, 최종 판단은 반드시 `판단 기록`에서 따로 남깁니다.

`검증 설계` 카드의 `검증 증거 코치`는 아래 항목을 자동 점검합니다.

| 항목 | 확인하는 내용 | 보강 행동 |
| --- | --- | --- |
| 문제 빈도 | 문제 강도와 반복 빈도 | 최근 30일 발생 횟수와 마지막 사례를 질문 |
| 실제 사례 | 추상 의견이 아닌 행동/사건 근거 | 인터뷰, 관찰, 외부 자료를 근거 노트로 저장 |
| 구매자와 지불 | 돈을 내는 주체와 예산/가격 단서 | 월 구독, 건당 비용, 절감액 기반 수수료 중 현실적인 모델 질문 |
| 도달 채널 | 이번 주 만날 수 있는 타겟 경로 | 연락 가능한 타겟 5명과 채널 기록 |
| 대안/경쟁 | 현재 해결 방식과 불만 | 엑셀, 카카오톡, 수동 처리, 경쟁 서비스 등 대안 3개 정리 |
| 행동 증거 | 클릭, 신청, 저장, 공유, 지불 의향 | 실험을 진행 중/완료로 바꾸고 결과 노트를 저장 |
| 리스크 수용 | 높은 리스크의 종료 또는 수용 조건 | 완화 방안, 차단 범위, 수용 조건 기록 |

`프롬프트 복사`는 다음 인터뷰/관찰 질문을 복사합니다. `근거 폼 채우기`는 같은 내용을 `근거 직접 기록` 폼에 넣어 바로 리서치 노트로 저장할 수 있게 합니다.

## 6. 리스크 추가

`리스크 추가` 영역에서 선택한 아이디어의 차단 요인을 기록합니다.

| 필드 | 예시 |
| --- | --- |
| 제목 | 개인정보 노출 가능성 |
| 영역 | 보안/개인정보 |
| 심각도 | 높음 |
| 완화 방안 | 민감정보 최소 수집, RLS 정책 확인, 감사 로그 저장, 사용자별 권한 분리 |

저장 후 `관련 리스크`에 표시됩니다. 리스크 상태는 `열림`, `완화 중`, `종료` 버튼으로 변경합니다.

## 7. 판단 기록

`판단 기록`은 점수보다 중요합니다. 점수는 추천이고, 판단 기록은 의사결정 로그입니다.

1. 점수와 증거 공백을 확인합니다.
2. `판단` 드롭다운에서 `진행`, `추가 조사`, `전환`, `중단` 중 하나를 고릅니다.
3. `판단 근거`에 왜 그렇게 결정했는지 적습니다.
4. `{판단} 기록` 버튼을 클릭합니다.

예시 판단 근거:

```text
센터장 인터뷰 3건에서 일정 누락과 보호자 문의 대응 시간이 반복 문제로 확인됨.
다만 건강정보 처리 리스크가 있어 MVP는 일정/메모 중심으로 제한하고,
개인 건강정보 업로드는 2차 단계로 보류한다.
```

## 8. 실험 계획 추가

`실험 계획`은 실제 개발 전에 가장 작은 검증 단위를 정하는 영역입니다.

| 필드 | 예시 |
| --- | --- |
| 실험 | 센터장 5명 대상 클릭 가능한 일정 관리 프로토타입 테스트 |
| 성공 지표 | 5명 중 3명 이상이 현재 운영 방식보다 낫다고 평가하고, 2명 이상이 월 3만원 지불 의향을 밝힘 |

실험 상태는 `계획`, `진행 중`, `완료`로 바꿀 수 있습니다. 실험을 완료한 뒤에는 `실험 결과 기록`에 결과, 배운 점, 다음 판단, 다음 행동을 저장합니다. 이 기록은 산출물 라이브러리에 `리서치 노트`로 남습니다.

## 9. 오케스트레이션 런북 만들기

`오케스트레이션 보드`에서 `런북 만들기`를 클릭하면 전체 역할 단계가 자동 생성됩니다.

생성되는 단계:

| 단계 | 목적 |
| --- | --- |
| 전략 | 기회, 판단 기준, 제약, 다음 실행 약속 정의 |
| 리서치 | 사용자 고통, 시장 수요, 경쟁, 규제 사실 검증 |
| 제품 | 증거를 PRD와 수용 기준으로 전환 |
| 디자인 | 흐름, 화면, 빈 상태, 사용성 리스크 정리 |
| 개발 | 가장 작은 유용한 프로토타입 구현 범위 정의 |
| QA | 핵심 여정과 회귀 위험 검증 |
| 디버깅 | 실패 재현, 원인 분리, 수정 검증 |
| 보안 | 개인정보, 비밀값, 권한, 악용 경로 검토 |
| 출시 | 진행/전환/중단/추가 조사 판단 |

각 단계에서 `템플릿 사용`을 누르면 해당 단계에 맞는 산출물 형식이 자동으로 채워집니다. 내용을 보완한 뒤 `산출물 저장`을 누르고, 상태를 `완료`로 바꾸면 출시 준비도에 반영됩니다.

## 10. 산출물 저장과 승인

워크벤치 하단에는 자동 생성되는 산출물이 있습니다.

| 출시 체크리스트 초안 | 출시 전 증거, 승인, 리스크, 실험, QA/보안 상태를 확인 |
| 출시 판단 패킷 | 출시 준비도, 개발 완료 게이트, 산출물 승인 큐, 리스크, 태스크 증거를 묶어 진행/추가 조사/전환/중단 권고를 생성 |
| 출시 후 학습 루프 | Day 7/14/30 지표, 이벤트, 인터뷰 질문, 다음 판단 기준을 출시 체크리스트 산출물로 저장 |
| 산출물 리뷰 메모 | 라이브러리에서 산출물 버전 변화, 섹션 변화, 리뷰 강도, 승인 전 확인 항목을 복사 |

`라이브러리` 하위 탭 상단의 `앱 제작 산출물 승인 큐`는 아이디어 브리프, 리서치 노트, PRD, MVP 명세, 백엔드 결정, 디자인 브리프, 기술 명세, 개발 런북, 출시 체크리스트를 순서대로 점검합니다. `다음 항목 열기`를 누르면 생성해야 할 화면이나 승인해야 할 라이브러리 항목으로 이동합니다. 큐에서 `없음`은 산출물 생성 필요, `초안`은 승인 대기, `승인`은 다음 게이트 통과 상태입니다.

테스트 순서:

1. `아이디어 브리프 초안`에서 `산출물 저장`을 누릅니다.
2. `리서치 브리프 초안`에서 인터뷰/경쟁/가격/규제 체크를 확인하고 `산출물 저장`을 누릅니다.
3. `7일 검증 스프린트`에서 모집 메시지와 Day 7 판정 기준을 확인하고 `산출물 저장`을 누릅니다.
4. `근거 직접 기록`에 인터뷰 메모나 외부 자료를 하나 입력하고 `근거 저장`을 누릅니다.
5. `검증 완료 요약`에서 PRD 진입 조건과 보류 조건을 확인하고 `산출물 저장`을 누릅니다.
6. `제품 산출물` 하위 탭으로 이동해 `PRD 진입 준비도`의 다음 보강 항목을 확인합니다.
7. `핸드오프 복사` 또는 `핸드오프 저장`으로 PRD 작성자에게 넘길 판단 메모를 남깁니다.
8. 준비도 카드의 `검증 산출물 보강`, `실험 확인`, `리스크 확인`, `판단 기록` 버튼이 올바른 영역으로 이동하는지 확인합니다.
9. `PRD 초안`에서 `산출물 저장`을 누릅니다.
10. `MVP 슬라이스 플랜`에서 Slice 0 수동 검증, Slice 1 얇은 제품, Slice 2 AI/자동화, Slice 3 출시 하드닝 순서가 맞는지 확인하고 `플랜 저장`을 누릅니다.
11. 같은 하위 탭에서 `MVP 명세 초안`을 저장합니다.
12. `앱 개발 프로세스`의 `준비와 산출물` 탭에서 백엔드 선택 스코어카드를 확인합니다.
13. `백엔드 실행 체크리스트`에서 권장 백엔드, 필수 환경변수, 권한 규칙, 로컬 검증, 프로덕션 게이트, 롤백 기준이 선택지별로 달라지는지 확인하고 `체크리스트 복사` 또는 `체크리스트 저장`을 누릅니다.
14. 빠른 테스트라면 `개발 패키지 저장`으로 백엔드 결정, 백엔드 실행 체크리스트, 디자인 브리프, 기술 명세, 개발 런북, 앱 블루프린트, MVP 스캐폴드 매니페스트를 한 번에 저장합니다.
15. `디자인 준비도`와 `개발 착수 준비도`가 어떤 항목에서 막히는지 확인합니다.
16. 같은 영역에서 `디자인 생성 프롬프트`, `앱 블루프린트`, `MVP 스캐폴드 매니페스트`를 복사하거나 저장한 뒤, 디자인 브리프, 기술 명세, 개발 런북, Codex 구현 핸드오프를 저장합니다. 개발 착수 준비도에 `환경변수 경계`, `백엔드 규칙 검증`, `롤백/배포 로그`가 표시되는지 확인합니다.
17. `완료와 핸드오프` 탭에서 `역할별 프롬프트 팩`을 복사하거나 저장해 전문 역할별 작업 지시가 같은 문맥을 공유하는지 확인합니다.
18. 디자인 브리프 저장 후 `디자인 준비도`가 갱신되는지 확인합니다.
19. `개발 태스크` 탭으로 이동해 `기본 태스크 생성`을 누릅니다.
20. 같은 화면의 `개발 킥오프 브리프`에서 다음 차단 항목, Slice 1 범위, 제외 범위, 완료 증거 형식을 확인하고 `브리프 저장`을 누릅니다.
21. `개발 실행 순서 게이트`에서 기획, 디자인, 데이터, 백엔드, 프론트, QA, 보안, 배포 순서와 시작 가능/대기 태스크가 나뉘는지 확인합니다.
22. `순서 복사` 또는 `순서 저장`으로 선행 조건 문서를 남깁니다.
23. 필요한 경우 `직접 태스크 추가`에서 버그, 디자인 수정, 배포 작업, 고객 검증 작업을 추가합니다.
24. `다음 개발 액션`에서 우선 실행 태스크가 올바르게 추천되는지 확인합니다.
25. `구현 실행 패키지`에서 현재 필터 태스크와 승인 산출물이 묶이는지 확인하고 `패키지 복사` 또는 `패키지 저장`을 누릅니다.
26. `티켓 복사` 또는 `백로그 복사`로 GitHub Issue, PR 본문, Codex 핸드오프에 붙일 실행 문서를 복사합니다.

## 10-1. 개발 태스크 보드 사용법

`앱 개발 프로세스`의 개발 태스크 보드는 산출물을 실제 구현 작업으로 바꾸는 영역입니다.

태스크를 만들기 전 `개발 킥오프 브리프`를 먼저 확인합니다. 이 브리프는 MVP 슬라이스 플랜을 기준으로 이번 개발 범위를 Slice 1 얇은 제품으로 제한하고, Slice 2 AI/자동화, 결제, 외부 계정 조작, 복잡한 관리자 화면처럼 아직 만들지 않을 범위를 명시합니다. 차단 항목이 남아 있으면 브리프 저장 후 필요한 산출물 승인, 백엔드 결정, 리스크 종료를 먼저 처리합니다.

| 열 | 의미 | 테스트 행동 |
| --- | --- | --- |
| 할 일 | 아직 시작하지 않은 구현 작업 | 기본 태스크 생성 직후 모든 항목이 여기에 생성되는지 확인 |
| 진행 중 | 현재 작업 중인 항목 | 우선순위 높은 항목 1개를 진행 중으로 변경 |
| 차단 | DB, 환경변수, 정책, 외부 확인 때문에 막힌 항목 | 필요한 경우 차단 사유를 증거에 기록 |
| 완료 | 수용 기준을 충족한 항목 | 커밋, 검증 명령, Preview URL, 스모크 결과를 증거에 저장 |

기본 태스크는 PRD/MVP 범위 잠금, 디자인 흐름, 백엔드 권한, 데이터 모델, 핵심 화면, 상태 UX, 실험 계측, 보안 점검, 배포 스모크로 구성됩니다. 배포 태스크에는 Preview/Production URL, Vercel inspect URL 또는 배포 로그, 환경변수 변경 여부, 롤백 기준을 증거에 남깁니다.

각 태스크 카드의 증거 입력란 아래에는 `증거 품질` 힌트가 표시됩니다. 예를 들어 백엔드 태스크는 커밋/검증 외에 `허용/차단`, `RLS` 또는 `Security Rules` 근거를 요구하고, 배포 태스크는 Preview/Production, Vercel 로그, 롤백 기준을 요구합니다. 점수가 낮아도 저장은 가능하지만, 완료 처리 전에는 빠진 힌트를 보강하는 것이 좋습니다.
태스크 보드 상단의 `증거 보강 우선순위`는 빠진 항목이 많은 태스크를 먼저 보여줍니다. 이 목록에서 상위 태스크부터 증거를 보강하면 개발 완료 보고서와 출시 준비도 검토가 훨씬 빨라집니다.
`차단` 상태 태스크가 있으면 `차단 해소 큐`가 나타납니다. 이 큐는 담당 역할, 다음 액션, 해소 증거, 에스컬레이션 기준을 보여주므로, 차단 사유를 증거에 적고 조건이 충족되면 `진행 중` 또는 `완료`로 옮기면 됩니다.
태스크가 많아지면 `태스크 필터`에서 상태, 담당, 증거 공백 여부를 좁혀 봅니다. 예를 들어 `차단` + `증거 공백 있음`을 선택하면 지금 해소해야 할 막힌 카드만 볼 수 있습니다.
`필터 복사`는 현재 필터 조건과 보이는 태스크 목록을 마크다운으로 복사합니다. Codex에게 특정 담당자의 막힌 작업만 넘기거나, 이슈 트래커에 좁혀진 백로그를 붙일 때 사용합니다.
`실행 프롬프트`는 같은 필터 결과를 구현 에이전트용 지시서로 복사합니다. 역할별 초점, 작업 목록, 차단 해소 기준, 검증 명령, 완료 보고 형식이 함께 포함됩니다.
옆의 `저장`은 현재 필터의 실행 프롬프트를 `개발 런북` 산출물로 보관합니다. 같은 필터 조건으로 여러 번 저장하면 산출물 버전이 올라갑니다.
`구현 실행 패키지`는 필터 실행 프롬프트보다 한 단계 더 큰 묶음입니다. 승인된 PRD, MVP 명세, 디자인/기술 산출물과 현재 필터 태스크, 남은 게이트, 실험, 리스크, 검증 명령을 합쳐서 실제 구현 세션의 시작 프롬프트로 쓸 수 있습니다.
`MVP 빌드 명령 패킷`은 구현 실행 패키지보다 더 최종적인 첫 구현 명령입니다. 앱 블루프린트, 스캐폴드 매니페스트, Codex 구현 핸드오프, 출시 판단 패킷을 합쳐서 새 저장소 또는 새 서비스 디렉터리의 첫 메시지로 넘길 수 있습니다.
`QA 검수 매트릭스`는 구현 완료 직후 실행할 핵심 여정, 권한/RLS 또는 Firebase Rules, 민감정보, 모바일, 오류 상태, 배포 스모크, 완료 보고 형식을 정리합니다.
산출물 라이브러리에서는 최근 개발 핸드오프가 상단에 요약됩니다. `필터 저장본` 배지가 붙은 개발 런북은 태스크 필터에서 만든 실행 프롬프트입니다.
라이브러리 필터는 유형, 상태, 출처를 함께 지원합니다. 필터 저장본만 찾으려면 출처를 `필터 실행 프롬프트`로 좁히면 됩니다.

자동 생성만으로 부족할 때는 `직접 태스크 추가`를 사용합니다.

| 필드 | 입력 예시 |
| --- | --- |
| 태스크 제목 | 모바일 입력 폼 버튼 겹침 수정 |
| 유형 | 프론트 |
| 우선순위 | 높음 |
| 담당 역할 | ux-polisher |
| 수용 기준 | 390px 모바일 폭에서 저장 버튼과 텍스트가 겹치지 않고, 저장 실패 메시지가 입력 영역 아래에 표시됨 |

`개발 완료 게이트`는 아래 조건을 확인합니다.

| 조건 | 통과 기준 |
| --- | --- |
| 개발 태스크 생성 | 선택한 아이디어에 구현 태스크가 1개 이상 있음 |
| 차단 태스크 없음 | `차단` 상태 태스크가 없음 |
| 모든 태스크 완료 | 모든 구현 태스크가 `완료` 상태 |
| 완료 증거 기록 | 완료된 모든 태스크에 증거가 있음 |
| QA와 보안 단계 완료 | QA와 보안 오케스트레이션이 모두 완료 상태 |

백엔드나 배포 태스크를 완료할 때는 Supabase RLS 또는 Firebase Security Rules/IAM의 허용/차단 검증, Vercel 환경변수 공개/서버 경계, 배포 로그 또는 inspect 링크가 함께 남아 있어야 합니다.

완료 증거는 아래처럼 짧게 남기면 충분합니다.

```text
커밋: 81a6fe8
검증: pnpm quality:full 통과, pnpm smoke:prod 통과
Preview/Production: https://ai-venture-lab.vercel.app
Vercel inspect: https://vercel.com/.../deployment-id
환경변수: NEXT_PUBLIC_SUPABASE_URL은 공개 키, SUPABASE_SERVICE_ROLE_KEY는 서버 전용. Production 재배포 확인.
백엔드 규칙: RLS 허용 - 본인 workspace idea insert 성공. RLS 차단 - 다른 workspace idea update 거부 확인.
롤백: 직전 Ready 배포 dpl_xxx로 복구, DB 변경 없음.
남은 리스크: 브라우저 자동 스모크는 아직 수동 확인.
```

Firebase를 선택한 앱이면 백엔드 규칙 줄을 이렇게 바꿉니다.

```text
백엔드 규칙: Security Rules 허용 - 로그인 사용자의 본인 문서 write 성공. 차단 - 다른 uid 문서 write 거부. App Check는 Preview에서 경고 없음.
```

게이트를 검토한 뒤 `보고서 저장`을 누르면 개발 완료 보고서가 산출물 라이브러리에 저장됩니다.
보고서에는 백엔드, 데이터, 보안, 배포 태스크의 `릴리스 증거 요약`이 함께 들어갑니다. 이 섹션에서 증거 품질 점수와 보강 필요 항목을 확인한 뒤 최종 출시 판단으로 넘어갑니다.

## 10-2. 학습 루프 이벤트 확인

`학습 루프` 메뉴는 출시 전후에 어떤 행동이 실제로 쌓였는지 확인하는 운영 대시보드입니다.

2단계부터는 내부 운영 이벤트뿐 아니라 실제 MVP나 외부 앱 서버에서 발생한 제품 사용 이벤트도 받을 수 있습니다. 외부 앱은 AI Venture Lab DB에 직접 접속하지 않고, 서버 전용 API로 이벤트를 보냅니다.

필요한 Production 환경변수:

| 환경변수 | 위치 | 용도 |
| --- | --- | --- |
| `SUPABASE_SERVICE_ROLE_KEY` | Vercel 서버 전용 | 수집 API가 RLS를 우회해 검증된 이벤트를 저장 |
| `TELEMETRY_INGEST_SECRET` | Vercel 서버 전용, 외부 MVP 서버에도 동일하게 보관 | 이벤트 수집 API 호출 인증 |

수집 API:

```http
POST https://ai-venture-lab.vercel.app/api/telemetry/ingest
Content-Type: application/json
Authorization: Bearer <TELEMETRY_INGEST_SECRET>
```

예시 payload:

```json
{
  "ideaId": "선택한 아이디어 ID",
  "eventName": "product_core_action",
  "eventCategory": "product",
  "source": "mvp-production",
  "anonymousId": "stable-user-or-device-id",
  "sessionId": "current-session-id",
  "properties": {
    "action": "created_first_record",
    "path": "/dashboard",
    "plan": "free"
  }
}
```

`anonymousId`, `sessionId`, `user-agent`는 API에서 해시되어 저장됩니다. `properties`에는 이메일, 전화번호, 이름, 계좌, 카드, 원문 대화 같은 직접 식별 정보를 넣지 않습니다.

자동 기록되는 주요 이벤트:

| 이벤트 | 언제 기록되는지 |
| --- | --- |
| 아이디어 생성/검증 패키지 저장 | 새 아이디어 저장 또는 발굴 후보 패키지 저장 |
| 점수/상태 저장 | 점수화 화면에서 단계, 판단, 점수, 증거를 저장 |
| 리스크/판단/실험 기록 | 리스크 추가, 판단 기록, 실험 추가/상태 변경 |
| 오케스트레이션 기록 | 런북 생성, 단계 추가, 단계 상태/산출물 저장 |
| 산출물 저장/승인 | 브리프, PRD, MVP, 개발/출시 산출물 저장 또는 상태 변경 |
| 개발 태스크 기록 | 태스크 생성, 상태 변경, 완료 증거 저장 |
| 제품 사용 기록 | 외부 MVP 서버가 보낸 화면 조회, 핵심 행동, 활성화, 결제 신호, 피드백, 오류, 이탈 신호 |

테스트 순서:

1. 점수 저장, 리스크 추가, 실험 추가, 산출물 저장 중 2개 이상을 수행합니다.
2. 왼쪽 실행 메뉴에서 `학습 루프`를 엽니다.
3. 최근 7일, 14일, 30일 이벤트 카운트가 표시되는지 확인합니다.
4. `최근 이벤트` 목록에서 이벤트 이름, 범주, 시각, 속성이 보이는지 확인합니다.
5. `학습 리포트 초안`을 검토하고 `리포트 저장`을 눌러 리서치 노트로 저장합니다.
6. `MVP 사용 이벤트 연결` 카드에서 API endpoint, 현재 아이디어 ID, payload 예시를 복사합니다.
7. 같은 카드의 `서버 환경변수`, `Next.js 서버 라우트`, `브라우저 helper`, `운영 스모크` 스니펫을 복사해 외부 MVP 저장소에 붙입니다.
8. 외부 MVP 서버에서 테스트 이벤트를 보낸 뒤 `제품 이벤트` 카운트와 최근 이벤트 목록이 갱신되는지 확인합니다.
9. `제품 이벤트 퍼널`에서 방문, 가입 시작, 가입 완료, 핵심 행동, 활성화, 결제 신호 단계별 건수와 전 단계 대비 전환율을 확인합니다.
10. `이벤트 택소노미`에서 아직 수집되지 않은 필수 이벤트가 무엇인지 확인하고, `퍼널 저장`으로 리서치 노트에 남깁니다.

운영 수집 API만 빠르게 검증하려면 아래 스모크를 사용합니다. 비밀값은 코드에 저장하지 말고 현재 터미널 환경변수로만 넣습니다.

```powershell
$env:TELEMETRY_INGEST_SECRET="Vercel에 등록한 TELEMETRY_INGEST_SECRET"
$env:TELEMETRY_SMOKE_IDEA_ID="학습 루프 카드나 Supabase ideas 테이블에서 복사한 idea id"
pnpm smoke:telemetry
```

성공하면 `Inserted telemetry event id`가 출력됩니다. 이후 `학습 루프`를 새로고침하고 `제품 이벤트` 카운트, `최근 이벤트`, `제품 이벤트 퍼널`의 `핵심 행동` 단계가 늘어났는지 확인합니다.

이벤트에는 본문 내용 전체를 저장하지 않고 유형, 상태, 길이, 개수 같은 운영 메타데이터만 저장합니다. 실제 사용자 행동 이벤트를 붙일 때도 같은 원칙을 유지해야 합니다.

운영 DB에는 `supabase/migrations/20260506010000_add_learning_telemetry.sql`와 동일한 테이블/인덱스/RLS 정책이 적용되어 있어야 합니다. 행동을 수행했는데 이벤트가 쌓이지 않으면 먼저 `telemetry_events` RLS insert/select 정책을 확인합니다.

## 11. 출시 준비도 100% 테스트 시나리오

아래 순서대로 수행하면 출시 준비도 게이트를 끝까지 테스트할 수 있습니다.

1. 새 아이디어를 입력합니다.
2. 점수 항목을 모두 입력하고 `점수 저장`을 클릭합니다.
3. 리스크를 1개 이상 추가합니다.
4. 높음/치명적 리스크가 있다면 상태를 `종료`로 변경합니다.
5. 실험 계획을 1개 추가합니다.
6. `런북 만들기`를 클릭합니다.
7. QA 단계와 보안 단계에 `템플릿 사용` 후 `산출물 저장`을 클릭합니다.
8. QA 단계와 보안 단계 상태를 `완료`로 변경합니다.
9. 실험 결과 기록에 결과, 배운 점, 다음 판단을 저장합니다.
10. 아이디어 브리프, 리서치 브리프, 7일 검증 스프린트, 수동 근거 기록, 검증 완료 요약, PRD 초안, MVP 슬라이스 플랜, MVP 명세 초안을 저장합니다.
11. 제품 산출물 탭의 `PRD 진입 준비도`가 어느 항목을 막고 있는지 확인합니다.
12. `앱 개발 프로세스`의 `준비와 산출물` 탭에서 백엔드 선택 스코어카드와 백엔드 실행 체크리스트를 확인하고 `개발 패키지 저장`으로 백엔드 결정, 백엔드 실행 체크리스트, 디자인 브리프, 기술 명세, 개발 런북, 앱 블루프린트, MVP 스캐폴드 매니페스트를 저장합니다.
13. `디자인 생성 프롬프트`가 제품 맥락, 화면 목록, 상태, 모바일, 권한, v0/Stitch 출력 지시를 포함하는지 확인합니다.
14. `앱 블루프린트`가 라우트, 컴포넌트, 데이터 모델, API/서버 액션, 권한, 수용 테스트를 포함하는지 확인합니다.
15. `MVP 스캐폴드 매니페스트`가 파일 트리, 환경변수, Supabase/Firebase 규칙 초안, 검증 명령을 포함하는지 확인합니다.
16. 백엔드 실행 체크리스트가 선택 백엔드에 맞는 환경변수, 권한 규칙, 허용/차단 검증, 프로덕션 게이트, 롤백 기준을 포함하는지 확인합니다.
17. `디자인 준비도`와 `개발 착수 준비도`가 저장/승인 상태에 따라 갱신되는지 확인합니다.
18. `개발 태스크` 탭에서 `기본 태스크 생성`을 누르고, 직접 태스크를 1개 추가해 저장합니다.
19. `개발 실행 순서 게이트`가 기획 완료 전에는 후속 태스크를 대기로 보여주고, 기획/데이터/백엔드 상태를 완료로 바꿀수록 시작 가능 태스크가 갱신되는지 확인합니다.
20. `다음 개발 액션` 추천을 확인하고 티켓 복사와 백로그 복사가 동작하는지 확인합니다.
21. 모든 개발 태스크를 `완료`로 변경한 뒤 각 태스크에 완료 증거를 저장합니다.
22. `완료와 핸드오프` 탭에서 `MVP 빌드 명령 패킷`을 복사하거나 저장합니다.
23. 같은 탭에서 `QA 검수 매트릭스`를 복사하거나 저장합니다.
24. 같은 탭에서 `개발 완료 게이트`가 100%인지 확인하고 개발 완료 보고서를 저장합니다.
25. 산출물 라이브러리의 `앱 제작 산출물 승인 큐`가 다음 승인 대상을 올바르게 보여주는지 확인합니다.
26. 산출물 리뷰 카드에서 리뷰 강도와 승인 전 확인 항목을 확인하고 `리뷰 메모`를 복사합니다.
27. 산출물 라이브러리에서 PRD, MVP 명세, 백엔드 결정, 디자인 브리프, 기술 명세, 개발 런북, 출시 체크리스트를 `승인됨`으로 변경합니다.
28. 판단을 `진행` 또는 `추가 조사`로 선택하고 판단 근거를 기록합니다.
29. `출시 준비도`에서 출시 판단 패킷의 추천 판단과 차단 항목을 확인합니다.
30. `패킷 저장`을 눌러 출시 체크리스트 산출물 버전으로 보관합니다.
31. `출시 후 학습 루프`를 저장해 출시 후 7/14/30일 검토 기준을 남깁니다.
32. 왼쪽 실행 메뉴의 `학습 루프`에서 최근 이벤트와 학습 리포트 초안이 생성되는지 확인합니다.
33. `판단 근거 채우기`를 눌러 판단 기록 폼에 패킷 요약이 채워지는지 확인합니다.

## 13. 테스트 완료 기준

최소 테스트 완료 기준:

1. 매직 링크 로그인이 된다.
2. 워크스페이스를 만들 수 있다.
3. 아이디어 발굴에서 후보를 추출하고 최근 추출 실행 메타데이터와 기존 유사 기록을 확인할 수 있다.
4. 추출 후보의 검증 패키지 준비도, 사업/개발 스코어카드, 민감정보 경고, 원문 근거를 확인한 뒤 아이디어, 리스크, 실험, 아이디어 브리프, 리서치 브리프, 7일 검증 스프린트까지 개별 또는 상위 후보 일괄 패키지로 저장할 수 있다.
5. 새 아이디어를 저장할 수 있다.
6. 워크벤치에서 내 아이디어가 `편집 가능`으로 보인다.
7. 점수 저장, 리스크 추가, 판단 기록, 실험 추가가 된다.
8. 실험 결과와 배운 점을 리서치 노트로 저장할 수 있다.
9. 런북 만들기 후 각 단계 상태와 산출물을 저장할 수 있다.
10. 리서치 브리프/7일 검증 스프린트/수동 근거 기록/실험 결과/검증 완료 요약/PRD/MVP/개발 패키지/출시 체크리스트 산출물이 라이브러리에 저장된다.
11. 앱 개발 탭의 백엔드 선택 스코어카드가 선택 아이디어 문맥에 따라 Supabase/Firebase/Firebase SQL Connect/Hybrid 점수를 보여준다.
12. 백엔드 실행 체크리스트를 복사하거나 백엔드 결정 산출물로 저장할 수 있다.
13. 앱 개발 탭이 `준비와 산출물`, `개발 태스크`, `완료와 핸드오프`로 분리되고 디자인/개발 착수 준비도가 저장/승인 상태에 맞게 갱신된다.
14. 디자인 생성 프롬프트를 복사하거나 디자인 브리프 산출물로 저장할 수 있다.
15. 앱 블루프린트를 복사하거나 기술 명세 산출물로 저장할 수 있다.
16. MVP 스캐폴드 매니페스트를 복사하거나 개발 런북 산출물로 저장할 수 있다.
17. 개발 태스크 보드에서 기본 태스크를 만들고 직접 태스크를 추가할 수 있다.
18. 개발 실행 순서 게이트가 바로 시작 가능한 태스크와 선행 조건 대기 태스크를 구분하고, 순서 문서를 복사/저장할 수 있다.
19. 다음 개발 액션 추천, 티켓 복사, 백로그 복사가 동작한다.
20. 개발 태스크 상태와 완료 증거를 저장할 수 있다.
21. 개발 완료 게이트와 개발 완료 보고서 저장이 동작한다.
22. MVP 빌드 명령 패킷을 복사하거나 개발 런북 산출물로 저장할 수 있다.
23. QA 검수 매트릭스를 복사하거나 개발 런북 산출물로 저장할 수 있다.
24. 좌측 `우선 검토 후보`가 저장/점수/리스크/실험/승인 상태 변화에 따라 갱신된다.
25. 산출물 라이브러리의 앱 제작 산출물 승인 큐에서 다음 생성/승인 항목과 전체 승인률을 확인할 수 있다.
