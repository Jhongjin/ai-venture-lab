# App Planning Principles

Use these as source-informed anchors when turning an idea into a PRD or MVP plan.

## Discovery

- Reframe requested solutions as problems to be solved.
- Learn the user's wider journey, current workaround, constraints, accessibility needs, and success measures.
- Do not start building during discovery unless the goal is a low-cost research artifact.

## Alpha or MVP

- Prototype only the riskiest or most valuable part of the journey.
- Test whether the service can meet user needs and be cost-effective before committing to beta-scale work.
- Consider offline, operational, and support channels, not only the screen flow.

## Startup Validation

- Treat the idea as a falsifiable hypothesis.
- Talk to users about real past behavior, not just opinions about a proposed feature.
- Prefer manual MVPs, concierge workflows, and payment/time-saving proof before broad automation.

## Shaping

- Set an appetite before scope grows.
- A shaped pitch needs a problem, appetite, solution sketch, rabbit holes, and no-gos.
- Hunt for design, technical, legal, operational, and data risks that could multiply delivery time.

## PRD Contract

- A useful PRD aligns product, design, engineering, QA, security, and launch around shared context.
- Include goals, assumptions, user stories, designs, success metrics, and out-of-scope items.
- Acceptance criteria should be testable.
